## Commit 3: `refactor: fix template flag detection and clean up dead code`

### C-1. 修复模板 flag 检测 bug

#### 当前 bug
`main.go:109-137` 使用 `cfg.HostTpl != defaults.Host` 判断用户是否显式传入 `--host-tpl`. 这是错误的: 如果用户显式传入的值恰好等于默认模板字符串, 在 `--mini` 模式下会被错误地覆盖为 Mini 模板.

根因: cobra 的 flag 默认值在 `init()` 中被设为 `defaults.XxxTpl`, 所以 "`cfg.XxxTpl != defaults.Xxx`" 近似于"用户显式设置过", 但在边角情况下失真.

#### 修复
改用 cobra 的官方 API `cmd.Flags().Changed(flagName)`, 它**只在用户显式传入时**返回 `true`, 依赖默认值时返回 `false`.

#### 具体改动
**`main.go`**
- `buildTemplates(isAll bool)` → `buildTemplates(cmd *cobra.Command, isAll bool)`
- 调用点 L74: `tpls := buildTemplates(cmd, isAll)`
- 替换 5 处条件:

```go
// Before
if cfg.HostTpl != defaults.Host {
    tpls.Host = cfg.HostTpl
} else if cfg.MiniStyle {
    tpls.Host = formatter.MiniHostTpl
}

// After
if cmd.Flags().Changed("host-tpl") {
    tpls.Host = cfg.HostTpl
} else if cfg.MiniStyle {
    tpls.Host = formatter.MiniHostTpl
}
```

同样方式处理 `cpu-tpl`, `mem-tpl`, `load-tpl`, `disk-tpl`.

**必须**: 函数体中原 L107 `defaults := formatter.DefaultTemplates()` 删除. 比较基准已从字符串值变为 "flag 是否被设置", 该变量不再被引用, 保留会导致 Go 编译失败 (declared and not used).

#### 新增测试
**`main_test.go`** (新文件, package main)
- `TestBuildTemplates_RespectsExplicitFlagInMiniMode` — bug 的直接回归测试
- `TestBuildTemplates_DefaultInMiniMode` — 保证不回退旧行为
- `TestBuildTemplates_DefaultNonMini` — 默认非 mini 路径
- `TestBuildTemplates_ExplicitNonMini` — 显式非 mini 路径

测试需要构造 `*cobra.Command` 并注册所有 5 个 tpl flag + `--mini`, 然后用 `cmd.ParseFlags([]string{...})` 模拟用户输入. 约 60-80 行.

### C-5. 移除 `Manager.mu`, 改为本地变量组装

#### 理由
当前 5 个 goroutine 写 `*types.Info` 的**不同字段**, Go 内存模型把每个 struct 字段视为独立内存位置, 不同字段的并发写本就不构成 race. 加 mutex 反而掩盖意图. 改造为"每个 goroutine 独立产出结果, 主 goroutine 用 `wg.Wait()` 建立 happens-before 后组装", 无锁, 语义更清晰.

#### 改动
**`internal/collector/collector.go`**

```go
// Manager struct 精简
type Manager struct {
    opts Options
    // 删除 info *types.Info, mu sync.Mutex
}

// NewManager 精简
func NewManager(opts Options) *Manager {
    return &Manager{opts: opts}
}

// Collect 改为本地装配
func (m *Manager) Collect(names []CollectorName) *types.Info {
    info := &types.Info{}
    var wg sync.WaitGroup
    for _, name := range names {
        wg.Add(1)
        go func(n CollectorName) {
            defer wg.Done()
            switch n {
            case NameHost:
                info.Host = m.collectHost()
            case NameCPU:
                info.CPU = m.collectCPU()
            case NameMem:
                info.Mem = m.collectMem()
            case NameLoad:
                info.Load = m.collectLoad()
            case NameDisk:
                info.Disk = m.collectDisk()
            }
        }(name)
    }
    wg.Wait()
    return info
}

// collectXxx 方法签名改为返回值
func (m *Manager) collectHost() *types.HostInfo { /* 原逻辑, 但返回 info 而不是写 m.info */ }
// ... CPU, Mem, Load, Disk 同理
```

#### 并发安全论证
Go 内存模型 ([golang.org/ref/mem](https://golang.org/ref/mem)) 明确: "Structure values... a struct field is a separate memory location". 多个 goroutine 并发写入同一 struct 的**不同字段**不构成 race (race detector 不会报警). `wg.Wait()` 在主 goroutine 对所有子 goroutine 的写入建立 happens-before, 所以返回的 `info` 读取是安全的.

| commit | 若发现问题 | 回滚方式 |

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| gopsutil v4 有未发现的字段变更 | 低 (agent 全量核查) | 中 (编译错误, 易修) | commit 前本地跑 build + test |
